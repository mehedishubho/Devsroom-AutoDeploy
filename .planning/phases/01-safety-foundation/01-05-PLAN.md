**Root cause (from debug session):** verify_deployment() at line 1380 only checks the main plugin file (`$plugin_path . '/' . $plugin_slug . '.php'`) for PHP syntax errors via token_get_all() at line 1408. A realistic "bad deploy" with syntax errors in any secondary PHP file passes verification silently.

**Fix 1: Recursive PHP syntax scanning in verify_deployment():**

After the existing Check 2 (main file syntax) passes, add a new check that recursively scans ALL PHP files:

```php
// Check 2b: Syntax check all PHP files recursively.
$php_files = new \RecursiveIteratorIterator(
    new \RecursiveDirectoryIterator($plugin_path, \RecursiveDirectoryIterator::SKIP_DOTS)
);
$syntax_errors = array();
foreach ($php_files as $file) {
    if ($file->isFile() && $file->getExtension() === 'php') {
        try {
            $code = file_get_contents($file->getRealPath());
            if (false !== $code) {
                token_get_all($code, TOKEN_PARSE);
            }
        } catch (\Throwable $e) {
            $syntax_errors[] = basename($file->getPathname()) . ': ' . $e->getMessage();
        }
    }
}
if (! empty($syntax_errors)) {
    $checks['syntax_all'] = false;
    return array(
        'success' => false,
        'message' => 'PHP syntax errors found: ' . implode('; ', array_slice($syntax_errors, 0, 3)),
        'checks'  => $checks,
    );
}
$checks['syntax_all'] = true;
```

Place this AFTER the existing Check 2 (main file syntax) at line 1417 and BEFORE Check 3 (plugin header) at line 1419. This ensures:
- Main file is checked first (existing behavior preserved)
- All other PHP files are checked recursively
- First 3 errors are reported in the message (avoid flooding)
- Uses RecursiveIteratorIterator + RecursiveDirectoryIterator (standard PHP SPL)
- SKIP_DOTS prevents scanning `.` and `..` entries

**Fix 2: Windows rollback cleanup in deploy() method:**

At lines 486-490 in the rollback section, the Windows fallback does:
```php
$this->copy_directory($old_path, $plugin_path);
$this->cleanup_dir($old_path);
```

The problem: `copy_directory()` overwrites but doesn't remove files that exist in `$plugin_path` but NOT in `$old_path`. This means broken files from the failed deployment remain.

Fix by adding `cleanup_dir($plugin_path)` BEFORE `copy_directory()`:
```php
if (! @rename($old_path, $plugin_path)) {
    // Windows fallback — clean target first, then copy from .old.
    $this->cleanup_dir($plugin_path);
    $this->copy_directory($old_path, $plugin_path);
    $this->cleanup_dir($old_path);
}
```

This ensures the broken deployment directory is fully removed before restoring from .old.
