**Root cause (from debug session):** `trigger_deployment()` calls `$deployment_manager->deploy()` synchronously, then does `wp_redirect()` and `exit`. The entire deployment runs in the same HTTP request. Lock is acquired at start of deploy() and released in finally block before deploy() returns. Page is never rendered while lock exists.

**Fix:** Replace synchronous `deploy()` call with `wp_schedule_single_event()` + immediate redirect. The scheduled event fires on next page load (via WP-Cron), at which point the lock is acquired and held while the admin page can render showing the lock indicator.

**Changes to admin/class-repository-manager.php:**

1. **Add WordPress hook registration in constructor** (or wherever actions are registered):
   ```php
   add_action('devsroom_autodeploy_async_deploy', array($this, 'handle_async_deployment'), 10, 2);
   ```
   This hook fires when the scheduled event triggers.

2. **Add `handle_async_deployment(int $repository_id, bool $activate_after)` method:**
   ```php
   public function handle_async_deployment(int $repository_id, bool $activate_after = false): void
   {
       $deployment_manager = Deployment_Manager::get_instance();
       $result = $deployment_manager->deploy($repository_id, 'manual', 0, true);

       if ($result['success'] && $activate_after) {
           $repository = $this->get_repository($repository_id);
           if ($repository) {
               $this->activate_plugin_by_slug($repository['plugin_slug']);
           }
       }
   }
   ```
   - No redirect — this runs in WP-Cron context, not HTTP context
   - No nonce check needed — scheduled events are internal WordPress operations
   - Handles both deploy and deploy+activate flows

3. **Modify `trigger_deployment()` method:**
   Replace the synchronous `$deployment_manager->deploy()` call + redirect logic with:
   ```php
   $activate_after = /* determine from caller — pass as parameter or check POST key */;
   wp_schedule_single_event(time(), 'devsroom_autodeploy_async_deploy', array($repository_id, $activate_after));

   if ($activate_after) {
       wp_redirect(admin_url('admin.php?page=devsroom-autodeploy-repositories&deploy_queued=activating'));
   } else {
       wp_redirect(admin_url('admin.php?page=devsroom-autodeploy-repositories&deploy_queued=true'));
   }
   exit;
   ```
   - Remove the entire block that calls `$deployment_manager->deploy()` synchronously (lines 364-389)
   - The `wp_schedule_single_event` stores the event in the options table — it fires on the next WordPress page load (any visitor or WP-Cron)
   - The lock is NOT acquired during this request — it's acquired when the scheduled event fires and calls `deploy()`

4. **Add "deployment queued" notice in the template:**
   In `admin/partials/repository-form.php`, add a notice for the `deploy_queued` query parameter:
   ```php
   if (isset($_GET['deploy_queued'])) {
       if ($_GET['deploy_queued'] === 'activating') {
           echo '<div class="notice notice-info"><p>' . esc_html__('Deployment queued. The plugin will be deployed and activated shortly.', 'devsroom-autodeploy') . '</p></div>';
       } else {
           echo '<div class="notice notice-info"><p>' . esc_html__('Deployment queued. It will start shortly.', 'devsroom-autodeploy') . '</p></div>';
       }
   }
   ```

**Important:** Do NOT modify webhook handler or polling scheduler — they remain synchronous. The lock visibility issue only affects the admin UI (manual deployments). Webhook/polling deployments run in background contexts where lock visibility doesn't matter.
